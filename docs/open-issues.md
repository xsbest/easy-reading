# Open Issues

- `expo-speech` 的可用 voice 由设备平台决定。本轮只能保证“尽量区分预设 + 失败时显式回退提示”，不能保证所有预设在所有设备上都映射到不同音色。
- 当前书架已挂接两本本地英文技术书的完整桌面 PDF，并把阅读页导读扩展到 18 页；但阅读器内仍然是导读分页，不是全文 PDF 内嵌排版结果。
- `Node.js Design Patterns` 与 `Programming TypeScript` 当前提供的是覆盖全书主线的章节导读 + AI 导读，用于功能联调，不是逐页还原原书全文渲染结果。
- “翻译本页”当前通过外跳 Google Translate Web 完成，未接入官方翻译 SDK 或内嵌插件。
- 原 PDF 入口依赖桌面文件路径；若源文件被移动、删除，阅读页里的“打开 PDF”会失效。
- ElevenLabs PoC 当前只覆盖中文预设，且直接依赖 `EXPO_PUBLIC_ELEVENLABS_API_KEY`；正式环境仍需要服务端代理和密钥保护。
- 当前 PoC 没有新增原生音频播放依赖，云端合成音频只在支持 `Audio` / `Blob` / `URL.createObjectURL` 的运行环境内直接播放；其他环境会自动回退 `expo-speech`。
- 已补充 `docs/voice-provider-research.md`：如果要明显提升“好听程度”和咬字准确性，需要批准替换当前 `expo-speech` 底层方案；仅调整预设表不足以解决核心问题。
- “把豆包语音包搞下来用”当前不能按 `expo-speech` 的系统 voice 模型直接实现；若继续推进，需要先确认豆包是否以云端 TTS 供应商方式接入，并批准替换当前朗读底层。
- 最新诉求已进一步收敛为“中文 / 英文男声都用豆包，女声都用秋芝”；该需求仍不能通过调整 `voicePresets` 直接实现，必须先确认豆包与秋芝的中英文 voice 能力，并批准云端 TTS 主链路改造。
- 在豆包能力未核实前，不能假设“秋芝”天然支持英文；如果秋芝仅支持中文，英文女声策略需要重新选型。
- 即使后续确认豆包 / 秋芝能力可用，也不能按“下载语音包”思路接回当前项目；实际落地形态仍是云端 TTS 合成音频 + 客户端播放。
- 若要满足“男声中英文都用豆包，女声都用秋芝”，至少需要新增音频播放能力，并评估是否保持 Expo Managed Workflow 下的 REST 接入，不宜在未审批前直接动 `src/` 实现。
- “书籍搜索功能”可先做本地即时搜索 MVP，但实现入口更接近书架页而非阅读页，已超出当前 `reader-bugfix` 默认实现范围，需要先批准扩 scope。
- 阅读页内容密度提升后，需要在小屏设备上重点复测边缘热区与正文滚动的冲突风险。
- 去掉卷页特效后，若用户仍觉得翻页反馈偏弱，下一步只应在位移时长和底页透明度上微调，不建议把卷角或书脊阴影加回来。
- 阅读主题切换当前为阅读页局部状态，关闭阅读页后不会持久保存，需要后续根据产品决策再决定是否升级到全局偏好。
- 赛博朋克主题依赖高对比与半透明面板，小屏和低亮度设备上需要补测可读性与眩光感是否过强。
- 新诉求“点击后布局不要被压缩，全部都以抽屉形式展开”已收敛为可行方向，但仍需先批准具体交互边界：推荐统一使用覆盖式抽屉，不让展开面板参与正文布局。
- 新诉求“书本内容不要停在 18 页，要全部加载完”当前不能直接等价为改大 `mockBooks.ts`；需先确认目标是更长章节摘要、原 PDF 全文渲染，还是 CDN 远程静态内容。
- 若目标是原 PDF 全文可读，在 Expo Managed Workflow 下大概率涉及新增 PDF 解析/展示能力，属于底层内容链路调整，未审批前不应直接改 `src/`。
- 若目标是把全文或章节切片丢到 CDN，需要先确认资源格式、缓存与版权边界；这不是当前文档里已批准的默认能力。
- 已补充 `docs/reader-drawer-and-content-research.md`：记录抽屉式覆盖展开与“18 页之外的全量内容加载”候选方案、依赖影响与取舍结论；其中已明确当前真正会压缩正文布局的是 `目录导读` 与 `AI 导读` 的内联展开，不是主题/语音托盘。
- 当前两本书在阅读器里都只有 18 页导读摘要，不是全文 PDF 拆页结果；若要满足“全部加载完放进去”，推荐优先审批“预处理后的长文本 / 静态章节资源”方案，再决定是否引入 CDN / 对象存储分发链路。
- 直接在 Expo 客户端运行时解析 PDF 以生成全文页不适合作为当前默认方案；该路径高概率需要新增依赖，并把需求扩大成底层能力改造。
