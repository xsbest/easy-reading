# Reader Drawer And Full-Content Research

## 背景

本次诉求聚焦两个点：

- 阅读页里点击展开的内容不要再挤压正文布局，统一改成抽屉式浮层展开
- 当前书本内容不要停留在 18 页导读，要评估“全部加载完放进去”的可行方式

本次仅输出候选方案与取舍结论，不直接改 `src/`。

## 当前现状

### 1. 布局压缩点

- `ReaderScreen.tsx` 里的主题托盘、语音托盘已经是悬浮浮层，不占正文布局
- `目录导读` 和 `AI 导读` 仍以内联面板插入 `ScrollView` 内容流，会把正文整体往下推
- 现状说明：现在不是“全部点击面板都会压缩布局”，而是阅读页里仍有两类面板没有抽屉化

### 2. 内容加载点

- `src/data/mockBooks.ts` 当前两本书都通过 `guidePages` 生成阅读页内容
- `nodeJsGuidePages` 当前是 18 页
- `programmingTypeScriptGuidePages` 当前也是 18 页
- 这 18 页属于导读摘要页，不是原 PDF 全文拆页结果

## 候选方案

### A. 阅读页全部改为抽屉式浮层

范围：

- 把 `目录导读`、`AI 导读` 从 `ScrollView` 内联卡片改成覆盖式抽屉
- 延续现有主题托盘、语音托盘的浮层方向，不新增导航层
- 抽屉默认覆盖在阅读壳层之上，关闭时不占正文高度

优点：

- 满足“点击后布局不要被压缩”
- 与当前主题/语音托盘实现方向一致，改动集中在 `ReaderScreen.tsx`
- 不需要新增依赖，也不需要替换底层能力

代价：

- 需要重新处理抽屉层级、滚动穿透、边缘翻页热区冲突
- 小屏下要重点验证抽屉高度与顶部/底部控制区的遮挡关系

结论：

- 这是推荐方案
- 依赖影响：不需要新增依赖
- 底层影响：不需要替换底层能力

### B. 保持内联面板，只继续收紧样式

范围：

- 继续压缩 `目录导读`、`AI 导读` 的内边距、字号和摘要长度
- 不改成交互抽屉

优点：

- 改动最小

缺点：

- 不能真正满足“不占用布局”
- 内容一旦展开，正文仍然会被推走

结论：

- 不推荐

### C. 全文内容改成本地预处理静态资源

范围：

- 在仓库外先把原 PDF 提取为结构化文本，再按阅读器需要切成更长页数组
- 产物可落为本地 JSON / TS 数据文件，再由当前阅读器读取

优点：

- 可以保持 Expo Managed Workflow，不必新增运行时依赖
- 客户端离线可读，性能和可控性高于运行时临时解析 PDF
- 不需要把 PDF 解析能力塞进当前 App

代价：

- 这不是“直接读完整 PDF”，而是“预处理后的全文文本导入”
- 仓库体积会明显增加
- 需要先确认版权、文本质量、切页规则和是否允许把全文落仓

结论：

- 这是“全文加载”方向里的首选实现
- 依赖影响：默认不需要新增客户端依赖
- 底层影响：不需要替换阅读器底层，但需要批准替换当前 mock 数据来源

### D. 全文内容改成远端 CDN / 对象存储分发

范围：

- 把预处理后的全文分页 JSON 或分章节资源放到 CDN / 对象存储
- 阅读页按书籍或章节拉取远端内容

优点：

- 不会把大体积全文内容直接塞进仓库
- 后续扩书更容易

代价：

- 需要远端存储、发布流程和缓存策略
- Expo 客户端需要补远端内容获取、失败回退和离线策略
- 若直接分发原文全文，版权与内容授权风险更高

结论：

- 可以作为后续扩书方案
- 不是当前最小落地解
- 依赖影响：客户端未必需要新三方库，但需要新增远端内容基础设施
- 底层影响：不需要替换阅读器底层能力，但需要批准新增内容分发链路

### E. 运行时直接解析 PDF 并在 App 里生成全文页

范围：

- 在 Expo 客户端侧直接解析 PDF 文本并分页

优点：

- 理论上最接近“全部加载完放进去”

代价：

- 当前项目没有 PDF 解析链路
- 高概率需要新增依赖，且 Expo Managed Workflow 下兼容性与性能风险都高
- 会把本轮需求从阅读页体验调整扩大成底层能力改造

结论：

- 明确不推荐作为本轮方案
- 依赖影响：大概率需要新增依赖
- 底层影响：等同于新增 PDF 解析底层能力

## 取舍结论

### 对“全部都以抽屉形式展开，不占用布局”

推荐采用方案 A：

- 只改 `ReaderScreen.tsx` 的局部交互层
- 不新增依赖
- 不替换底层能力
- 批准后可按现有浮层托盘风格统一到 `目录导读`、`AI 导读` 和其他后续阅读辅助面板

### 对“书本内容不要只加载 18 页，要全部加载完”

推荐采用方案 C，方案 D 作为后续扩展：

- 先确认是否允许把全文预处理后导入项目
- 如果允许，优先走“离线预处理全文 -> 静态分页资源”的方案
- 如果不适合落仓，再评估 CDN / 对象存储分发

不推荐直接做方案 E：

- 当前项目未具备客户端 PDF 解析底座
- 会超出本张 change card 的“未批准前不直接改代码”边界

## 审批前结论

- 抽屉化：可以做，不需要新增依赖，不需要替换底层能力
- 全文加载：不能直接在当前实现里假设“18 页导读 = 已加载全书”
- 若要把全文真正接入阅读器，至少需要先批准以下一项：
  - 批准把预处理后的全文静态资源纳入仓库
  - 或批准新增远端 CDN / 对象存储内容分发链路
- 在审批通过前，不应直接修改 `src/` 去伪装成“已经支持全文加载”
